# PROYECTO: Aplicación para Estudios de Mercado Inmobiliario

## OBJETIVO
Tener una aplicación que concentre la información necesaria para crear estudios de mercado, la información proviene de diferentes fuentes de datos como INEGI, Banxico, Sectur, Datatur, Datamexico, CONAPO, entre otras. La aplicación deberá mostrar gráficas correspondientes a los indicadores que se ocupan en los estudios de mercado, algunos a nivel nacioanl, otros a nivel estatal, otros a nivel municipal, otros a nivel de localidad (pueblos) y otros a nivel de ciudades importantes dentro de México.

## ESTRUCTURA GENERAL DEL PROYECTO
Deberá tener una etapa para ingesta de datos que se ejecutará de manera diaria, mensual, trimestral, semestral, anual y quinquenalmente de acuerdo a la periodicidad con que se actualice cada fuente de datos y alimentar a la base de datos; una etapa para procesamiento y limpieza de datos, y una etapa para carga en base de datos (PostgreSQL). Una API deberá leer la base de datos y llevar la información hacia el frontend. La aplicación está estandarizada con UV (gestión de dependencias) y Ruff (linting/formato). Ver `.cursor/rules/uv-ruff-standards.mdc` y ejecutar `ruff check . --fix && ruff format .` antes de commit.

## FRONTEND
Deberá contar con una sección que pida usuario (PhiQus) y contraseña (estudiosdemercado) para que el usuario pueda ingresar a la interfaz principal. La tipografía general de la app será personalizada (assets/ballingermono-light.ttf) y la paleta de colores de toda la app (que se irá aplicando a cada gráfica de forma progresiva dependiendo de la cantidad de elementos que requiera cada gráfica) será (#0576F3, #36F48C, #F47806, #F3F40B, #F479F4). La interfaz principal deberá tener en la parte superior izquierda un logo (assets/logo.png) y en seguida del logo, el título "Aplicación para estudios de Mercado". Debajo deberá mostrarse una sección para ajustes de estilo, que le permita al usuario modificar el tamaño de la tipografía (títulos, datos del eje x, datos del eje y) en las gráficas, cambiar entre la fuente por defecto o un set de fuentes de Google (que entre otras incluya la tipografía comfortaa), y una sección que le permita al usuario introducir una paleta de colores totalmente nueva (en formato HEX) o regresar a la paleta de colores por defecto. Debajo, deberá contar con una pestaña para cada nivel en el que se agruparán lás gráficas (Nacional, Estatal, Ciudades, Municipios, Localidades), debajo de la zona para visualizar cada gráfica deberá haber un botón para analizar con IA y debajo del botón deberá aparecer la tabla de datos correspondiente a la gráfica que se esté mostrando.

## ANÁLISIS CON IA
El botón para analizar con IA deberá enviar un JSON al modelo de lenguaje (llama-3.3-70b-versatile) utilizando la API KEY configurada en las variables de entorno (`GROQ_API_KEY`), el modelo realizará la interpretación de los datos y buscará patrones, tendencias, outliers y dirá el por qué ocurre ese comportamiento, para lo cual deberá realizar una investigación en internet y emitir su juicio relacionando los datos de la tabla con el contexto del país. Su respuesta deberá estar en Español de México.

## INSTRUCCIONES OBLIGATORIAS
Cada vez que generes, modifiques o refactorices código relacionado con visualización de datos o gráficas, DEBES cumplir estrictamente con los siguientes requisitos de diseño y funcionalidad. NO sobrescribas estas reglas.

### 0. Variables
El proyecto se desplegará en un vps, por lo que se deberán usar rutas relativas para todos los recursos que ocupe la app. Todos los tokens, API KEYS y contraseñas deberán guardarse como variables de entorno y leerse de esa manera para no exponerlas en el código.

### 1. Interactividad y Funcionalidad
* **Interactividad:** La gráfica debe ser tan interactive como la librería y los datos lo permitan (tooltips, zoom, selección, etc.).
* **Exportación:** Debe incluir una funcionalidad nativa o botón para descargar la visualización como archivo **.PNG de alta resolución**.

### 2. Diseño y Estilos (Layout)
* **Colores** Recuerda pasarle la paleta de colores (#0576F3, #36F48C, #F47806, #F3F40B, #F479F4) ya definida a los elementos (barras, líneas, segmentos, leyendas) de cada gráfica.
* **Dimensiones y resolución** Siempre configura el área de las gráficas con una relación de aspecto 16:9 a 720p (1280x720) pero que la gráfica se adapte al tamaño de la pantalla del dispositivo en que se consulta.
* **Título:** Debe estar siempre **centrado** en la parte superior.
* **Leyenda (Si aplica):** Debe posicionarse **centrada en la parte inferior** de la gráfica.
* **Fuente de Datos:** Debe incluirse una leyenda de texto en la **parte inferior izquierda** citando la fuente oficial de los datos.

### 3. Tipografía
* **Fuente Global:** Todo el texto visible en la gráfica debe usar la familia tipográfica **"Aptos Light"**.
* **Tamaño:** El tamaño base de la fuente debe ser **14px** (o su equivalente en unidades relativas).

### 4. Origen de los datos
* **PostgreSQL** Toda la información que se muestre en las gráficas debe provenir de la base de datos propia del proyecto, en caso de que no se encuentre ahí, hay que verificar el ETL desde la ingesta de datos hasta la inyección de los mismos en PostgreSQL y corregir lo que sea necesario únicasmente para garantizar que el proceso se cumpla.
* **API** Una vez confirmado que la información se encuentra en la base de datos, ajusta la API para que pueda enviarle esa data al frontend.

### 6. Tablas de datos
Debajo del botón de Analizar con IA (o de la respuesta del agente cuando ya se haya generado) deberá haber una tabla descargable como archivo .csv que contenga los mismo datos que se están mostrando en la gráfica.

### 7. Información sobre fuentes de datos
En la parte inferior de la app deberá haber un texto que indique la fecha en que se actualizó el ETL por última vez y cuántas fuentes se actualizaron correcta e incorrectamente en la última actualización.
